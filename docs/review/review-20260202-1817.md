# Review Report

## 摘要
本次审核对比了完整PRD与后端PRD在“后端逻辑”上的一致性。发现2处明确不一致点，涉及数据模型字段含义与配置变更生效规则。因文档较长，问题清单可能未穷尽。

## 问题清单
- 高：`total_task_count` 字段含义在两份文档中不一致。完整PRD将其定义为“业务系统数量”`docs/任务自动下发配置-PRD-20260129120000.md:903`，而后端PRD定义为“拆分后的实际任务数”`docs/任务自动下发配置-PRD-后端-20260129120000.md:739`。同时完整PRD任务拆分逻辑明确“单个业务系统可能拆分为多个任务”`docs/任务自动下发配置-PRD-20260129120000.md:437`，与“业务系统数量”定义相冲突。
- 中：配置变更生效规则在完整PRD内部与后端PRD之间存在冲突。完整PRD在配置变更处理处说明“业务系统范围保存时同步更新pending，资产范围执行时实时查询”`docs/任务自动下发配置-PRD-20260129120000.md:284`、`docs/任务自动下发配置-PRD-20260129120000.md:289`，但在边界处理又写“修改业务系统范围/资产范围当前轮次按原配置执行”`docs/任务自动下发配置-PRD-20260129120000.md:1007`、`docs/任务自动下发配置-PRD-20260129120000.md:1008`。后端PRD采用“同步更新pending/执行时实时查询”的口径`docs/任务自动下发配置-PRD-后端-20260129120000.md:848`、`docs/任务自动下发配置-PRD-后端-20260129120000.md:849`。当前两份文档口径不一致，且完整PRD内部自相矛盾。

## 修改建议
- 针对“total_task_count”定义：统一为“拆分后的实际任务数”或“业务系统数量”之一，并据此同步修正另一份文档及相关说明（如接口/统计含义）。若采用“实际任务数”，建议在完整PRD数据模型处改写并补充与拆分逻辑的对应关系。
- 针对“配置变更生效规则”：在完整PRD中二选一并删除另一处相反表述；同时在后端PRD补充一句“当前轮次”是否等同于“pending轮次”以消除歧义。

## 风险/待确认点
- 需产品/后端确认“total_task_count”是用于“任务粒度统计”还是“业务系统粒度统计”，以免影响轮次列表展示与统计指标。
- 需确认“配置变更是否影响已创建的pending轮次”是强一致需求还是仅影响下一轮次，以免出现调度时目标系统与资产范围不一致的问题。
